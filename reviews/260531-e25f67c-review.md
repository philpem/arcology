## Bug: Template crash for anonymous users on restricted artefacts (High)

**File:** `myapp/templates/artefacts/view.html:118,129`

The download-button block calls `current_user.can_bypass_all_restrictions()` directly
without first checking `current_user.is_authenticated`:

```jinja
{% if artefact.restrictions %}
    {% if current_user.can_bypass_all_restrictions(artefact.restrictions) %}
```

`AnonymousUserMixin` (what `current_user` is for unauthenticated visitors) has no such
method. If PUBLIC_MODE is on and an artefact that has restrictions is publicly visible,
any anonymous page view will raise `AttributeError` → 500.

The fix landed in the Python-side helpers (`_check_download_restrictions` etc.) but was
not applied to the template itself. Both call sites need the same guard:

```jinja
{% if current_user.is_authenticated and current_user.can_bypass_all_restrictions(artefact.restrictions) %}
```

The same pattern is needed at line 129 for `artefact_file_restrictions`.

**Remediation:** Add `current_user.is_authenticated and` before both
`can_bypass_all_restrictions` calls in the template.

## UX: Download button shown to anonymous users when PUBLIC_DOWNLOADS=False (Medium)

**File:** `myapp/templates/artefacts/view.html:139-141`

When `PUBLIC_DOWNLOADS=False` and `PUBLIC_MODE=True`, a non-restricted artefact still
renders a green "Download" button for anonymous visitors. Clicking it triggers a redirect
to the login page (the `@public_downloadable` route decorator handles this correctly), but
the misleading affordance is confusing UX.

The template has no awareness of the `PUBLIC_DOWNLOADS` flag. The context processor in
`app.py` already injects `public_mode`; it would need to also inject `public_downloads`
(or the template could read `config.PUBLIC_DOWNLOADS`).

**Remediation:** Inject `public_downloads` from the context processor, then wrap the
unrestricted download button:

```jinja
{% if current_user.is_authenticated or public_downloads %}
<a href="{{ artefact_url(artefact, 'download') }}" class="btn btn-success">
    <i class="bi bi-download"></i> Download
</a>
{% else %}
<a href="{{ url_for('login', next=request.url) }}" class="btn btn-outline-secondary">
    <i class="bi bi-box-arrow-in-right"></i> Log in to Download
</a>
{% endif %}
```

## Heading hierarchy: admin page uses `<h2>` instead of `<h1>` (Medium)

**File:** `myapp/templates/admin/index.html:5`

```html
<h2>Administration</h2>
```

Every other top-level page (`dashboard.html`, `items/index.html`, `taxonomy/index.html`,
`search/index.html`, `analysis/index.html`) uses `<h1>` as the page heading. The admin
page is the only outlier. This breaks the heading outline for screen readers and is
visually inconsistent (renders smaller than it should).

**Remediation:** Change to `<h1>Administration</h1>`.

## Accessibility: Navbar toggler missing `aria-label` (Low–Medium)

**File:** `myapp/templates/_base.html:30`

```html
<button class="navbar-toggler" type="button" data-bs-toggle="collapse" data-bs-target="#navbarNav">
    <span class="navbar-toggler-icon"></span>
</button>
```

The hamburger button has no accessible label. Bootstrap's own documentation shows
`aria-label="Toggle navigation"` on this element; without it screen readers announce an
unlabelled button.

**Remediation:**

```html
<button class="navbar-toggler" type="button" data-bs-toggle="collapse"
        data-bs-target="#navbarNav" aria-label="Toggle navigation"
        aria-controls="navbarNav" aria-expanded="false">
```

## Accessibility: Icon-only action buttons lack `aria-label` (Low–Medium)

**Files:** `artefacts/view.html:151`, `artefacts/_sidebar.html` (several), `items/view.html:93`

The delete button at `artefacts/view.html:151` is:

```jinja
<button type="submit" class="btn btn-outline-danger"><i class="bi bi-trash"></i></button>
```

No text or `aria-label`. Screen readers will announce "button" with no context.

Similarly, "Compute Hashes" and "Rescan Known Files" in `_sidebar.html` are icon-only.

**Remediation:** Add `aria-label` to every icon-only interactive element:

```jinja
<button type="submit" class="btn btn-outline-danger" aria-label="Delete artefact">
    <i class="bi bi-trash"></i>
</button>
```

## Inconsistency: Depth-indent uses inline `style=` instead of CSS custom property (Low)

**Files:** `myapp/templates/items/view.html:61`, `myapp/templates/items/index.html:20`,
`myapp/templates/artefacts/_sidebar.html:208`

All three use:

```jinja
style="padding-left: {{ 1 + depth * 1.5 }}rem;"
```

This bakes the formula into every rendered element and makes it impossible to adjust
spacing from a stylesheet. A CSS custom property would be cleaner and more maintainable:

```css
/* in base styles or a page <style> block */
.depth-indent { padding-left: calc(1rem + (var(--depth, 0) * 1.5rem)); }
```

```jinja
<td class="depth-indent" style="--depth: {{ depth }};">
```

## Inconsistency: Empty-state wording varies across templates (Low)

Different templates use different copy for the same concept:

| Template | Text |
|----------|------|
| `items/index.html` | "No items found." |
| `dashboard.html` | "No items yet." |
| `taxonomy/platforms.html` | "No platforms defined." |
| `taxonomy/tags.html` | "No tags defined." |
| `profile/index.html` | "No active API keys." |

No single form is wrong, but they are inconsistent. A project-wide convention
("No X yet." for lists that are empty by default; "No X found." for filtered/search
results that returned nothing) would be cleaner.

## Inconsistency: Required-field markers absent from most forms (Low)

Forms rendered manually (rather than via Bootstrap-Flask's `render_form()` macro)
do not mark required fields. `items/form.html`, `artefacts/upload.html`, and
`artefacts/edit.html` have required fields with no visual indicator.

**Remediation:** Either switch to `render_form()` (which handles this automatically)
or add `<span class="text-danger ms-1" aria-hidden="true">*</span>` after required
field labels, with a `<small class="text-muted">* required</small>` footnote.

---

## Consistency: Breadcrumbs absent from taxonomy and analysis list pages (Low)

Artefact and item detail pages have breadcrumbs. List-level pages for taxonomy
(`taxonomy/index.html`, `taxonomy/platforms.html`, etc.) and the analysis queue
(`analysis/index.html`) do not. Adding even a single-level breadcrumb ("Home › Taxonomy")
would be consistent and aids orientation for users arriving from search results.
